# Breakpoints

The Zen Design System uses a mobile-first approach with five breakpoints that enable responsive design across all device sizes. Every visual prop in the system can accept responsive values that adapt to different screen widths.

## Mobile-First Approach

Zen Design follows a **mobile-first** philosophy. This means:

1. **Base styles** are designed for mobile devices (0px+)
2. **Larger breakpoints** progressively enhance the experience
3. **CSS media queries** use `min-width`, not `max-width`

### How It Works

```css
/* Base styles apply to all screen sizes */
.zen-{component-name} {
  padding: var(--space-2);
}

/* sm breakpoint and above (640px+) */
@media (min-width: 640px) {
  .zen-{component-name} {
    padding: var(--space-4);
  }
}

/* lg breakpoint and above (1280px+) */
@media (min-width: 1280px) {
  .zen-{component-name} {
    padding: var(--space-6);
  }
}
```

In this example:

- **Mobile (0-639px)**: Uses `var(--space-2)`
- **Tablet (640px-1279px)**: Uses `var(--space-4)`
- **Desktop (1280px+)**: Uses `var(--space-6)`

## Using Breakpoints in CSS

### CSS Custom Properties

Breakpoint values are available as CSS custom properties in `tokens.css`:

```css
:root {
  --breakpoint-xs: 480px;
  --breakpoint-sm: 640px;
  --breakpoint-md: 1024px;
  --breakpoint-lg: 1280px;
}
```

**Important Note**: CSS media queries cannot directly use CSS custom properties. You must use the hardcoded pixel values in `@media` queries. However, the CSS custom properties are available for JavaScript usage and documentation.

### Media Query Pattern

Always use `min-width` with hardcoded pixel values:

```css
/* ✅ Correct */
@media (min-width: 640px) {
  .zen-{component-name} {
    /* styles */
  }
}

/* ❌ Incorrect - CSS variables don't work in media queries */
@media (min-width: var(--breakpoint-sm)) {
  .zen-{component-name} {
    /* styles */
  }
}
```

When using breakpoints in CSS, add comments referencing the token:

```css
@media (min-width: 640px) {
  /* --breakpoint-sm: 640px */
  .zen-component {
    /* responsive styles */
  }
}
```

## Responsive Props

Every visual prop in the design system can accept responsive values. This allows you to change styles at different breakpoints without writing custom CSS.

### Basic Syntax

Any prop that accepts a value `T` can also accept a responsive object:

```tsx
// Simple value (applies to all breakpoints)
<Component padding="md" />

// Responsive object (different values per breakpoint)
<Component padding={{ base: 'xs', sm: 'md', lg: 'xl' }} />
```

### Type Definition

```typescript
type Responsive<T> =
  | T
  | {
      base?: T
      xs?: T
      sm?: T
      md?: T
      lg?: T
    }
```

### Examples

**Example 1: Progressive Enhancement**

```tsx
<Component padding={{ base: 'xs', sm: 'md', lg: 'xl' }} />
```

This applies:

- `xs` padding on mobile (base)
- `md` padding from `sm` breakpoint (640px+) and up
- `xl` padding from `lg` breakpoint (1280px+) and up

**Example 2: Skipping Breakpoints**

```tsx
<Component padding={{ base: 'xs', lg: 'xl' }} />
```

This applies:

- `xs` padding from mobile up to 1279px
- `xl` padding from `lg` breakpoint (1280px+) and up

**Example 3: Starting at a Breakpoint**

```tsx
<Component gap={{ base: 'none', sm: 'md' }} />
```

This applies:

- No gap on mobile
- `md` gap from `sm` breakpoint (640px+) and up

**Example 4: Component with Multiple Responsive Props**

Components can support multiple responsive props:

```tsx
<Component
  padding={{ base: 'sm', md: 'lg' }}
  paddingHorizontal={{ base: 'xs', md: 'md' }}
  paddingVertical={{ base: 'sm', md: 'lg' }}
>
  Content
</Component>
```

This applies:

- `sm` padding, `xs` horizontal padding, `sm` vertical padding on mobile
- `md` horizontal padding from `md` breakpoint (1024px+) and up
- `lg` padding and `lg` vertical padding from `md` breakpoint (1024px+) and up

## How Responsive Props Work

### Behind the Scenes

When you provide a responsive prop value:

1. **Component processes the value**: Detects if it's a simple value or responsive object
2. **CSS custom properties are set**: Creates variables like `--{component-name}-{prop-name}-base`, `--{component-name}-{prop-name}-sm`, etc.
   - TypeScript: Uses `processResponsiveProps('{component-name}', { ... })` where `{component-name}` is kebab-case WITHOUT `zen-` prefix
   - SCSS: Uses `@include responsive-prop('zen-{component-name}', '{prop-name}', '{css-property}')` where class name includes `zen-` prefix (mixin strips it automatically)
3. **Class names are maintained**: For responsive objects, the base value's class name is added (e.g., `zen-{component-name}--{prop-name}-{base-value}`) to maintain compatibility with existing CSS
4. **Data attributes are added**: Adds `data-has-responsive-{prop-name}="true"` and breakpoint-specific attributes (e.g., `data-{prop-name}-{breakpoint}="{value}"`) to enable CSS rules
   - For camelCase props: Data attributes use kebab-case (e.g., `gridTemplateColumns` → `data-has-responsive-grid-template-columns`)
5. **CSS media queries apply**: Uses the CSS custom properties with fallback chains

### CSS Custom Property Fallback Chains

Responsive props use nested CSS variable fallbacks to create cascading values. Each breakpoint includes fallbacks for all previous breakpoints:

```css
@media (min-width: 480px) {
  /* --breakpoint-xs: 480px */
  .zen-{component-name}[data-has-responsive-{prop-name}='true'] {
    {css-property}: var(
      --{component-name}-{prop-name}-xs,
      var(--{component-name}-{prop-name}-base)
    );
  }
}

@media (min-width: 640px) {
  /* --breakpoint-sm: 640px */
  .zen-{component-name}[data-has-responsive-{prop-name}='true'] {
    {css-property}: var(
      --{component-name}-{prop-name}-sm,
      var(
        --{component-name}-{prop-name}-xs,
        var(--{component-name}-{prop-name}-base)
      )
    );
  }
}

@media (min-width: 1024px) {
  /* --breakpoint-md: 1024px */
  .zen-{component-name}[data-has-responsive-{prop-name}='true'] {
    {css-property}: var(
      --{component-name}-{prop-name}-md,
      var(
        --{component-name}-{prop-name}-sm,
        var(
          --{component-name}-{prop-name}-xs,
          var(--{component-name}-{prop-name}-base)
        )
      )
    );
  }
}

@media (min-width: 1280px) {
  /* --breakpoint-lg: 1280px */
  .zen-{component-name}[data-has-responsive-{prop-name}='true'] {
    {css-property}: var(
      --{component-name}-{prop-name}-lg,
      var(
        --{component-name}-{prop-name}-md,
        var(
          --{component-name}-{prop-name}-sm,
          var(
            --{component-name}-{prop-name}-xs,
            var(--{component-name}-{prop-name}-base)
          )
        )
      )
    );
  }
}
```

**How the fallback works:**

At each breakpoint, the browser tries fallbacks in order:

1. **xs breakpoint**: Try `--{component-name}-{prop-name}-xs`, fall back to `base`
2. **sm breakpoint**: Try `sm`, then `xs`, then `base`
3. **md breakpoint**: Try `md`, then `sm`, then `xs`, then `base`
4. **lg breakpoint**: Try `lg`, then `md`, then `sm`, then `xs`, then `base`

This ensures that:

- If you only set `base` and `lg`, intermediate breakpoints inherit from `base`
- Missing breakpoint values gracefully fall back through all smaller breakpoints
- The system works with partial breakpoint specifications

## Breakpoint Inheritance

When you don't specify a value for a breakpoint, the system inherits from the next smaller breakpoint:

```tsx
// You specify:
padding={{ base: 'xs', lg: 'xl' }}

// System behavior:
// 0px - 1279px:     Uses 'xs' (from base)
// 1280px and up:    Uses 'xl' (from lg)
```

The inheritance order is:

1. Check if current breakpoint value exists
2. If not, check next smaller breakpoint
3. Continue until `base` is found
4. If `base` doesn't exist, property is invalid

## Common Patterns

### Mobile vs Desktop Split

The most common pattern is to have different values for mobile and desktop:

```tsx
<Component padding={{ base: 'sm', md: 'xl' }} gap={{ base: 'xs', md: 'lg' }} />
```

### Three-Tier Responsive

For more granular control:

```tsx
<Component
  direction={{ base: 'column', sm: 'row' }}
  gap={{ base: 'xs', sm: 'md', lg: 'xl' }}
/>
```

### Progressive Enhancement

Start minimal and enhance at larger breakpoints:

```tsx
<Component columns={{ base: 1, sm: 2, lg: 4 }} gap={{ base: 'sm', md: 'md' }} />
```

### Nested Components with Responsive Props

Components can be nested and each can have its own responsive props:

```tsx
<Component columns={{ base: 1, md: 3 }} gap={{ base: 'sm', md: 'md' }}>
  <Component.Cell
    padding={{ base: 'sm', md: 'lg' }}
    paddingHorizontal={{ base: 'xs', md: 'md' }}
    paddingVertical={{ base: 'sm', md: 'lg' }}
  >
    Content adapts to screen size
  </Component.Cell>
</Component>
```

This pattern allows nested components to independently adapt their internal layout and spacing at different breakpoints.

## Best Practices

### When to Use Responsive Props

**✅ Do use responsive props for:**

- Layout properties (padding, margin, gap)
- Spacing (padding, margin)
- Grid and flex properties (columns, gap, direction)
- Visual sizing that needs to adapt to screen size

**❌ Don't use responsive props for:**

- Event handlers (`onClick`, `onChange`, etc.)
- React refs
- Content (`children`)
- Non-visual props

### Mobile-First Thinking

Always think about the mobile experience first:

```tsx
// ✅ Good: Mobile-first thinking
<Component padding={{ base: 'xs', lg: 'xl' }} />

// ⚠️ Less ideal: Desktop-first thinking
// (Works, but goes against mobile-first philosophy)
<Component padding={{ base: 'xl', md: 'lg', sm: 'md' }} />
```

### Performance Considerations

Responsive props use CSS custom properties and media queries, which are:

- **Highly performant**: No JavaScript runtime overhead
- **CSS-native**: Handled entirely by the browser
- **Cacheable**: Standard CSS that browsers optimize well

## TypeScript Support

All responsive props are fully typed for excellent developer experience:

```tsx
import type { Responsive } from 'zen-design'

interface MyComponentProps {
  padding?: Responsive<'xs' | 'sm' | 'md' | 'lg' | 'xl'>
  gap?: Responsive<string>
}
```

The `Responsive<T>` type ensures:

- Type safety for breakpoint values
- Autocomplete for breakpoint names
- Compile-time checking for valid values

## Technical Details

### CSS Custom Property Naming

Responsive props generate CSS custom properties with this pattern:

```
--{component-name}-{prop-name}-{breakpoint}
```

**Component name rules:**

- Component name should be kebab-case (e.g., `component-name`)
- In TypeScript: Pass component name WITHOUT `zen-` prefix (e.g., `'component-name'`)
- In SCSS: Pass full class name WITH `zen-` prefix (e.g., `'zen-component-name'`), the mixin automatically strips it

**Prop name rules:**

- Simple props: Use prop name as-is (e.g., `padding`, `gap`)
- CamelCase props: Preserve camelCase in CSS custom properties (e.g., `gridTemplateColumns` → `--{component-name}-gridTemplateColumns-base`)

Examples:

- `--{component-name}-{prop-name}-base`
- `--{component-name}-{prop-name}-xs`
- `--{component-name}-{prop-name}-sm`
- `--{component-name}-gridTemplateRows-base` (camelCase preserved)
- `--{component-name}-gridTemplateColumns-md` (camelCase preserved)

### Data Attributes

Components add data attributes to enable conditional CSS:

**Pattern:**

```
data-has-responsive-{prop-name}="true"
```

**Rules:**

- Simple props: Use prop name as-is (e.g., `data-has-responsive-padding`, `data-has-responsive-gap`)
- CamelCase props: Convert to kebab-case (e.g., `gridTemplateColumns` → `data-has-responsive-grid-template-columns`)
- Breakpoint-specific attributes: `data-{prop-name}-{breakpoint}="{value}"` (e.g., `data-gap-base="sm"`, `data-gap-md="md"`)

Examples:

- `data-has-responsive-{prop-name}="true"` when `{prop-name}` is responsive
- `data-has-responsive-grid-template-columns="true"` when `gridTemplateColumns` is responsive (camelCase → kebab-case)
- `data-{prop-name}-base="{value}"` for breakpoint-specific values

This ensures CSS rules only apply when responsive props are actually used, preventing undefined CSS variable warnings.

### Browser Support

Responsive props work in all modern browsers that support:

- CSS Custom Properties (CSS Variables)
- CSS Media Queries
- CSS `var()` function with fallbacks

This includes all major browsers from the last 5+ years.
